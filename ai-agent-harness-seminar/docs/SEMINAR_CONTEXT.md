# Seminar Context

이 문서는 프로젝트의 **Canonical Context**다. 다른 문서의 정의·범위·잠정 가정이 이 문서와 충돌하면 이 문서를 우선하되, 충돌 자체를 DECISION_LOG.md에 기록한다.

## 상태 표기

- **확인된 사실:** 저장소나 원출처로 현재 검증된 내용
- **잠정 가정:** 기획 진행을 위해 임시로 채택했으며 확인 후 바뀔 수 있는 내용
- **해석:** 확인된 사실을 특정 맥락에 적용한 설명
- **권고:** 선택 가능한 대안 중 현재 근거로 선호하는 방향

## 프로젝트 배경

**확인된 사실:** 이 저장소는 여러 세미나를 관리하는 Monorepo이며 본 프로젝트는 ai-agent-harness-seminar/ 아래에 격리된다. 앞선 발표의 주제는 Self-Attention, Large Language Model, KV Cache다.

**해석:** 앞선 발표가 모델 내부 계산과 추론 효율을 설명했다면, 후속 발표는 모델 밖의 상태·도구·권한·검증을 포함하는 시스템 관점으로 이동해야 한다.

## 세미나 목적

회사 내부 엔지니어가 AI Agent를 “더 똑똑한 챗봇”으로 오해하지 않고, Agent Loop와 Harness의 통제 요소를 설명하며, 검증 가능한 작은 업무 후보를 안전하게 설계하도록 돕는다. 특정 제품 도입이나 생산성 향상을 약속하는 자리가 아니다.

## 대상 청중의 특성

- 하드웨어 개발, Edge Computing, 진동진단, 플라즈마 시뮬레이션 실무자
- Python, C/C++, 데이터 분석, 계측, 실험 또는 시뮬레이션에 익숙할 가능성이 높음
- 일반 LLM Chat 경험은 있을 수 있으나 Agent 관련 용어는 익숙하지 않을 수 있음
- 추상 설명보다 파일·코드·데이터·실행 결과로 이어지는 사례가 필요함
- 이해에 그치지 않고 작고 되돌릴 수 있는 첫 시도를 설계해야 함

청중의 실제 숙련도와 참석 인원은 **알 수 없음**이며 사전 설문 결과에 따라 설명 깊이를 조정한다.

## 중심 명제

**설계상 주장:** Agent는 모델이 현재 상태를 관찰하고, 다음 행동을 선택하고, 도구를 사용하고, 결과를 다시 확인하며 목표를 향해 반복적으로 작업하는 시스템이다.

**설계상 주장:** Agent Harness는 Agent가 볼 수 있는 것, 할 수 있는 것, 유지하는 상태, 결과 검증, 종료 조건, 사람 승인이 필요한 작업을 통제한다.

이 문장은 이 세미나의 작업 정의다. 업계 전체가 합의한 단일 표준 정의라고 주장하지 않는다.

## 용어 정의

| 용어 | 이 세미나의 작업 정의 |
| --- | --- |
| LLM Chat | 사용자의 메시지에 모델이 텍스트 중심 응답을 생성하는 상호작용. 외부 행동 실행은 기본 전제가 아니다. |
| Tool Use | 모델 또는 애플리케이션이 명시된 Tool을 한 번 이상 호출해 외부 정보를 읽거나 행동을 수행하는 능력. 반복적 자율성까지 뜻하지 않는다. |
| Workflow | 단계·분기·오류 처리가 주로 코드나 규칙으로 미리 정해진 실행 흐름. 일부 단계에 모델을 포함할 수 있다. |
| Agent | 목표와 관측 결과를 바탕으로 다음 행동과 Tool 선택을 동적으로 결정하고, 결과를 재관찰하는 시스템. 자율성의 범위는 Harness가 제한한다. |
| Agent Loop | Observe → Decide → Act → Verify를 종료 조건까지 반복하는 제어 흐름. |
| Agent Harness | Context, Tool, State, Permission, Sandbox, Approval, Validator, Trace와 종료 조건을 묶어 Agent 실행을 통제하는 시스템 계층. |
| Tool | 파일 읽기, 코드 실행, 검색, API 호출처럼 모델 밖의 관찰 또는 행동을 제공하는 인터페이스. |
| Context | 현재 판단에 모델이 참조할 수 있도록 제공된 지시, 대화, 파일 일부, Tool 결과 등의 정보. |
| State | 반복 사이에 유지되는 작업 진행 정보. Context에 보이는 부분과 외부 저장소에 있는 부분을 구분한다. |
| Artifact | 코드, 보고서, 그림, 구조화 데이터 등 작업이 남긴 검사 가능한 산출물. |
| Validator | 완료 조건을 모델의 자기 선언과 독립적으로 판정하는 테스트, 스키마 검사, 수치 검사 또는 사람 검토. |
| Trace | 각 단계의 입력, 선택, Tool 호출, 결과, 승인과 오류를 시간 순서로 재구성할 수 있는 기록. |
| Guardrail | 허용 범위, 정책, 필터, 검증 등 위험 행동이나 잘못된 결과 가능성을 줄이는 통제. 완전한 안전 보장은 아니다. |
| Sandbox | 파일, 프로세스, 네트워크 등 실행 자원을 격리하거나 제한하는 환경. |
| Approval Gate | 지정된 위험 행동 전에 사람 또는 별도 권한 주체의 명시적 승인을 요구하는 경계. |

## 엔지니어링 폐루프 비유

Task/Goal은 목표값, 파일·Tool Output은 센서 관측, LLM 판단은 제어 로직, Shell·Simulator는 액추에이터, Validator는 시험 판정, Permission·Sandbox·Approval은 Safety Interlock, Trace는 Telemetry에 대응시켜 설명한다.

**비유의 한계:** LLM 기반 Agent는 결정론적 제어기가 아니다. 동일 입력에서도 경로가 달라질 수 있고, 자연어 목표는 제어 목표보다 모호할 수 있다. 따라서 안정성이나 실시간 제어 성능이 이 비유만으로 성립하지 않는다.

**잠정 가정:** 총 75분이며 Q&A 포함 여부는 미확정이다. 발표 시간이 바뀌면 데모와 슬라이드 수를 먼저 재산정한다.

## 주요 데모 후보

1. **메인 — 합성 진동 신호 분석:** metadata.yaml의 sampling frequency가 빠졌을 때 추정하지 않고 중단하며, 보완 후 metrics·waveform·spectrum·report·trace를 생성하고 Validator로 검사한다.
2. **보조 — 합성 Build/Test Failure 조사:** 작은 가짜 코드 저장소에서 실패 테스트를 재현하고 허용 파일만 최소 수정한 뒤 테스트로 완료를 판정한다.

두 데모 모두 설계 단계이며 실제 구현 여부와 실행 시간은 **알 수 없음**이다.

## 직군별 사용 사례

| 직군 | 후보 작업 | Agent보다 다른 대안이 나을 조건 | 검증 예 |
| --- | --- | --- | --- |
| 하드웨어 | 데이터시트·시험 결과 간 불일치 후보 정리 | 필드 매핑이 고정이면 Script | 필수 필드·단위·원문 링크 검사 |
| Edge Computing | 합성 빌드 실패 재현과 작은 패치 제안 | 원인이 규칙화되면 CI Workflow | 빌드·테스트·허용 경로 Diff |
| 진동진단 | 합성 신호 품질 검사와 분석 보고서 초안 | 고정 신호 처리면 Script | 샘플 수·주파수축·수치 재계산 |
| 플라즈마 시뮬레이션 | 설정 파일과 결과 Artifact 누락 조사 | sweep 경로가 고정이면 Workflow | 스키마·보존량·실행 로그 검사 |

진단이나 물리적 결론은 전문가 승인을 대체하지 않는다.

## 세미나 이후 사용 전환 전략

1. 반복 업무를 적되 Do Nothing, Chat, Script, Workflow도 함께 비교한다.
2. 실패 비용이 낮고 입력 복사본으로 실행 가능한 작은 후보 하나를 고른다.
3. Task Contract에 경로·도구·금지·Artifact·Validator·중단 조건을 쓴다.
4. 읽기 전용 또는 제한된 Sandbox에서 Trace를 수집한다.
5. 기존 방식과 정확성·검토 비용·총 소요 시간을 같은 기준으로 측정한다.
6. 근거가 있을 때만 제한적으로 확대한다. 이점이 없거나 검증 비용이 크면 중단한다.

## 보안 제약

- 회사 내부 정보와 실제 업무 데이터 사용 금지
- 합성 데이터만 저장소에 포함
- Token, 계정, 내부 주소, 실제 로그, 비공개 코드 저장 금지
- 외부 전송·네트워크·설치·삭제·생산/장비 접근은 기본 차단 또는 승인
- 입력 원본은 읽기 전용, 산출물은 제한된 경로에 기록
- 위험 작업은 Human Approval과 Rollback 계획 필요

## 성공 기준

- 참가자가 네 가지 실행 형태를 사례로 구분한다.
- Harness의 핵심 통제를 자신의 말로 설명한다.
- 최소 한 개의 Task Contract 또는 Use Case Canvas 초안을 만든다.
- 완료 선언과 Validator 판정을 구분한다.
- Agent 보류 조건과 Do Nothing 대안을 말할 수 있다.

실제 달성률 목표와 생산성 효과 수치는 근거가 없어 **알 수 없음**이다. 사전·사후 문항과 산출물 리뷰 기준을 정한 뒤에만 측정한다.
