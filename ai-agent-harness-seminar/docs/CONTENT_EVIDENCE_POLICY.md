# 콘텐츠 근거 및 출처 정책

## 목적

발표의 사실 주장과 설계상 해석을 섞지 않고, 청중이 근거 강도와 유효기간을 판단할 수 있게 한다.

## 출처 우선순위

1. 표준 원문, 공식 사양, 제품 공식 문서, 원 연구 논문과 원 데이터
2. 유지관리 주체의 저장소·릴리스 노트·보안 권고
3. 재현 가능한 독립 연구 또는 기술 보고서
4. 해설 자료는 맥락 보조에만 사용

검색 결과 요약, 마케팅 페이지, 출처 없는 블로그만으로 기술 사실을 확정하지 않는다.

## 주장 유형 표기

| 유형 | 의미 | 발표/문서 표기 |
| --- | --- | --- |
| 사실 | 원출처 또는 재현으로 확인 | 사실, 출처 각주 |
| 해석 | 사실을 세미나 맥락에 적용 | 해석 |
| 비유 | 이해를 위한 대응 관계 | 비유 및 한계 |
| 가정 | 아직 확인되지 않은 설계 입력 | 잠정 가정 |
| 권고 | 대안 비교 뒤 제안 | 권고와 철회 조건 |

## 출처 기록 형식

각 근거 항목은 가능한 한 다음을 기록한다.

- 주장 ID와 슬라이드 번호
- 문서/논문/표준명
- 저자 또는 관리 주체
- 직접 URL 또는 DOI
- 버전·릴리스·페이지/섹션
- 확인 날짜(YYYY-MM-DD)
- 뒷받침하는 주장과 적용 한계
- 재검증 날짜 또는 사건

예시:

    E-SLIDEV-001 | Slidev Syntax Guide | Slidev Team
    https://sli.dev/guide/syntax
    확인: 2026-07-21 | 대상: presenter notes와 Mermaid 지원
    한계: 제품 기능은 업데이트될 수 있어 발표 직전 재확인

## 빠르게 변하는 SDK·제품 정보

버전과 확인 날짜를 함께 표시하고, 발표 직전 공식 문서와 릴리스 노트에서 다시 확인한다. “현재”, “지원한다”, “기본값” 같은 표현에는 확인 시점을 붙인다. 과거 Trace나 캡처는 당시 버전을 명시한다.

## 수치 사용 규칙

- 계산식, 단위, 표본, 측정 조건, 출처를 확인할 수 있을 때만 사용한다.
- 검증 불가 수치는 “알 수 없음”으로 표기한다.
- 합성 수치는 “예시” 또는 “합성”으로 명확히 표시하고 실제 효과로 일반화하지 않는다.
- 평균만 제시하지 않고 가능하면 분산, 실패, 제외 기준과 비교 기준을 함께 기록한다.

## 생산성 향상 주장

Agent가 생산성을 높인다는 주장은 통제된 비교 기준, 작업 유형, 표본, 품질 기준, 사람 검토 시간, 실패·재작업, 비용을 함께 측정한 근거가 있을 때만 사용한다. 벤더 자체 연구는 연구 설계와 이해관계를 명시한다. 본 세미나 초기 자료에는 정량 효과 주장을 넣지 않는다.

## 인용·로고·캡처·외부 이미지

- 직접 인용은 필요한 최소 길이로 제한하고 따옴표와 출처를 표시한다.
- 제품 로고는 공식 브랜드 가이드와 사용 허가를 확인한다.
- 화면 캡처는 버전·날짜를 기록하고 계정, 내부 주소, Token, 실제 데이터가 보이지 않게 한다.
- 외부 이미지는 저작자, 원 URL, 라이선스, 변경 여부와 확인 날짜를 기록한다.
- 라이선스를 확인하지 못한 이미지는 사용하지 않고 자체 도형 또는 합성 이미지로 대체한다.

## 검증하지 못한 내용

슬라이드에는 “[근거 검증 필요]”, 문서에는 “Pending evidence”로 표시한다. 확인 전에는 Verified 상태로 바꾸지 않으며, 불확실성을 숨기기 위해 단정 표현을 사용하지 않는다.

## 제품 비교 중립성

동일한 작업, 권한, 데이터, Validator, 비용 범위를 적용한다. 기능 목록과 실제 검증 결과를 구분하고, 유료/무료·로컬/클라우드·데이터 보존 조건을 함께 기록한다. 특정 제품의 데모 성공을 Agent 일반의 우월성으로 확대하지 않는다. 비교 불가능한 항목은 순위를 만들지 않는다.

## 발표 직전 재검증

- SDK·CLI·모델·제품 버전과 지원 기능
- 모든 URL과 인용 문맥
- 수치·단위·그래프 재계산
- 로고·이미지 라이선스
- 캡처의 민감정보
- 데모 명령, Sandbox, Approval, 대체 Trace
- Storyboard의 Pending evidence
- GitHub Pages 공개 범위와 base path

## 판단 철회 조건과 검토일

공식 문서 변경, 원 연구의 철회, 재현 실패, 제품 버전 불일치, 라이선스 불명확이 발견되면 관련 주장을 제거하거나 Pending으로 되돌린다. 이 정책은 2026-07-21 작성되었으며 본편 콘텐츠 확정 전과 발표 7일 이내에 다시 검토한다.
