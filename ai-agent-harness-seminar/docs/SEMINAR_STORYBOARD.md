# Seminar Storyboard

본편 **28장 잠정 구성**이다. 최종 발표 문구가 아니라 슬라이드별 역할과 근거 요구를 정의한다. 시간 합계는 약 65분이며 나머지 10분을 전환·질문·지연 여유로 둔다. 총 75분과 Q&A 포함 여부는 미확정이다.

상태: **Proposed**는 구성 제안, **Verified**는 근거와 화면을 확인한 항목, **Pending**은 선행 결정이나 근거가 필요한 항목이다.

## Act 1. 모델에서 시스템으로

### 01. 모델에서 시스템으로

- **작업 제목:** From Models to Systems
- **슬라이드 목적:** 세미나의 질문과 범위를 한 문장으로 제시
- **핵심 메시지:** 모델 응답에서 검증 가능한 작업 시스템으로 관점을 이동한다.
- **화면 주요 내용:** 한/영 제목, Self-Attention → LLM → KV Cache → Agent System 연결선
- **시각자료:** 모델 경계가 Tool·State·Validator까지 확장되는 단순 도식
- **발표자 설명:** 특정 제품 소개나 생산성 약속이 아니라 시스템 구성과 적용 판단을 다룬다.
- **예상 시간:** 1.5분
- **근거/출처:** 앞선 세미나 실제 자료 확인 필요; Canonical Context
- **상태:** Pending

### 02. 앞선 세미나가 답한 것

- **작업 제목:** 모델 안에서는 무엇이 일어나는가
- **슬라이드 목적:** 기존 지식과 새 주제의 경계를 명확히 함
- **핵심 메시지:** 추론 구조를 이해해도 파일 수정과 실행 검증은 자동으로 생기지 않는다.
- **화면 주요 내용:** Self-Attention, LLM, KV Cache의 기존 질문과 이번 질문 비교
- **시각자료:** 모델 내부/외부 경계선
- **발표자 설명:** 앞선 내용의 세부 정의는 실제 발표 자료와 맞춘다.
- **예상 시간:** 2분
- **근거/출처:** 기존 발표 제목·범위 원자료 [Pending evidence]
- **상태:** Pending

### 03. 답변과 작업의 차이

- **작업 제목:** “방법을 설명”과 “산출물을 생성·검증”은 다르다
- **슬라이드 목적:** Agent 시스템이 필요한 문제를 구체화
- **핵심 메시지:** 작업에는 관찰, 행동, 상태 변화, 외부 판정이 필요하다.
- **화면 주요 내용:** Chat 응답 vs 파일 읽기→실행→Artifact→Test
- **시각자료:** 두 개의 짧은 sequence
- **발표자 설명:** Tool 호출 한 번을 곧바로 Agent라고 부르지 않는 작업 정의를 소개한다.
- **예상 시간:** 2.5분
- **근거/출처:** 세미나 작업 정의; 업계 표준 정의로 제시하지 않음
- **상태:** Proposed

### 04. “완료했습니다”의 한계

- **작업 제목:** 완료 선언은 Acceptance Test가 아니다
- **슬라이드 목적:** 외부 Validator의 필요성을 직관적으로 제시
- **핵심 메시지:** 텍스트 선언보다 Artifact, Test, Diff, Trace를 확인한다.
- **화면 주요 내용:** 완료 문장에 물음표, 오른쪽에 검사 가능한 증거
- **시각자료:** Claim → Evidence gate
- **발표자 설명:** 모델의 선의나 자신감이 아니라 시스템 판정 경계를 설계한다.
- **예상 시간:** 2분
- **근거/출처:** 시스템 설계 권고; 실증 효과 수치 사용 안 함
- **상태:** Proposed

## Act 2. Agent란 무엇인가

### 05. Chat, Tool Use, Workflow, Agent

- **작업 제목:** 네 실행 형태를 같은 축에서 비교
- **슬라이드 목적:** 흔히 섞이는 용어를 구분
- **핵심 메시지:** Tool 보유 여부와 다음 경로를 동적으로 선택하는지는 다른 축이다.
- **화면 주요 내용:** 경로 결정 주체, 반복, 외부 행동, 적합 업무 비교표
- **시각자료:** 2축 배치 또는 4열 카드
- **발표자 설명:** 분류는 이 세미나의 작업 정의이며 제품 이름과 동일시하지 않는다.
- **예상 시간:** 4분
- **근거/출처:** Canonical Context; 관련 제품 공식 정의 비교 [Pending evidence]
- **상태:** Pending

### 06. Agent Loop

- **작업 제목:** Observe → Decide → Act → Verify
- **슬라이드 목적:** Agent의 반복 구조를 설명
- **핵심 메시지:** Tool 결과를 다시 관찰하고 다음 행동을 바꾸는 폐루프가 핵심이다.
- **화면 주요 내용:** Goal, Context, Tool, Artifact, Validator, Stop
- **시각자료:** AgentLoopDiagram 컴포넌트
- **발표자 설명:** Verify가 모델 자기평가만을 뜻하지 않음을 강조한다.
- **예상 시간:** 3분
- **근거/출처:** 세미나 작업 정의; Agent 연구·공식 문서 매핑 [Pending evidence]
- **상태:** Proposed

### 07. 폐루프 제어 비유

- **작업 제목:** Sensor, Controller, Actuator, Interlock
- **슬라이드 목적:** 엔지니어 직군의 기존 개념과 연결
- **핵심 메시지:** 관측·행동·피드백·안전 경계라는 구조적 유사성이 있다.
- **화면 주요 내용:** 엔지니어링 시스템 ↔ Agent 시스템 대응
- **시각자료:** 좌우 대응 블록
- **발표자 설명:** 비유는 이해 도구일 뿐 동등성 주장이 아니다.
- **예상 시간:** 3분
- **근거/출처:** Canonical Context의 비유
- **상태:** Proposed

### 08. 비유가 깨지는 지점

- **작업 제목:** LLM은 결정론적 제어기가 아니다
- **슬라이드 목적:** 비유의 과잉 일반화를 차단
- **핵심 메시지:** 같은 입력에서도 경로가 달라질 수 있어 검증·제한·중단이 필요하다.
- **화면 주요 내용:** 동일 Goal에서 서로 다른 두 Trace, 동일 Validator
- **시각자료:** 분기 경로와 공통 gate
- **발표자 설명:** 변동성의 정량 크기는 모델·설정·업무별 측정 없이는 알 수 없다.
- **예상 시간:** 2분
- **근거/출처:** 선택한 실제 도구의 재현 실험 [Pending evidence]
- **상태:** Pending

### 09. Agent를 쓰지 않아야 할 때

- **작업 제목:** 유연성이 필요 없으면 단순한 도구가 낫다
- **슬라이드 목적:** Agent 만능론 방지
- **핵심 메시지:** 고정 경로·명확한 I/O·코드화 가능한 예외는 Script/Workflow 후보이다.
- **화면 주요 내용:** Do Nothing, Chat, Script, Workflow, Agent 선택 조건
- **시각자료:** 결정 트리
- **발표자 설명:** 검토 비용이 직접 작업 비용보다 큰 경우도 도입하지 않는다.
- **예상 시간:** 3분
- **근거/출처:** 설계 권고; 파일럿 측정으로 반증 가능
- **상태:** Proposed

### 10. 도입을 보류해야 할 때

- **작업 제목:** 검증 불가·고위험·비가역이면 멈춘다
- **슬라이드 목적:** 보류 조건을 분명히 함
- **핵심 메시지:** Validator와 안전 경계가 없으면 자율성을 추가하지 않는다.
- **화면 주요 내용:** 실시간 장비, 기밀 정책 부재, 큰 손실, 비가역 작업
- **시각자료:** Stop 표식이 있는 위험 목록
- **발표자 설명:** Agent를 사람의 안전 판단이나 실시간 제어 경로에 직접 연결하지 않는다.
- **예상 시간:** 2.5분
- **근거/출처:** 조직 보안·안전 정책 [Pending evidence]
- **상태:** Pending

## Act 3. Harness란 무엇인가

- **작업 제목:** 모델 밖에서 무엇을 통제하는가
- **슬라이드 목적:** Harness의 전체 구조 소개
- **핵심 메시지:** Context·Tool·State·Permission·Validator·Trace·Stop을 시스템이 통제한다.
- **화면 주요 내용:** Agent Loop를 둘러싼 Harness 박스
- **시각자료:** 경계형 아키텍처
- **발표자 설명:** Harness는 특정 단일 제품명이 아니라 세미나의 시스템 계층 정의다.
- **예상 시간:** 3분

### 12. Context, State, Artifact

- **작업 제목:** 보이는 정보, 유지되는 상태, 남는 산출물
- **슬라이드 목적:** 세 개념의 혼동 해소
- **핵심 메시지:** Context는 판단 입력, State는 지속 정보, Artifact는 검사 대상이다.
- **화면 주요 내용:** 수명주기와 저장 위치 비교
- **시각자료:** 세 개의 저장소 아이콘과 흐름
- **발표자 설명:** 같은 파일이 상황에 따라 State와 Artifact 역할을 모두 가질 수 있음을 설명한다.
- **예상 시간:** 2.5분
- **근거/출처:** 세미나 작업 정의
- **상태:** Proposed

### 13. Tool과 Permission

- **슬라이드 목적:** 최소 권한 원칙 설명

- **화면 주요 내용:** Read, Write, Execute, Network 권한 매트릭스

### 14. Validator와 종료 조건

- **작업 제목:** 누가 완료를 판정하는가
- **슬라이드 목적:** 독립 판정과 무한 반복 방지
- **핵심 메시지:** Acceptance Test, 재시도 한도, Stop Condition이 Agent 선언보다 우선한다.
- **화면 주요 내용:** 성공·재시도·승인·중단 네 경로
- **시각자료:** Validator 분기
- **발표자 설명:** Validator도 불완전할 수 있어 검사 범위와 거짓 통과 조건을 문서화한다.
- **예상 시간:** 3분
- **근거/출처:** 데모 Acceptance Criteria와 Validator 설계
- **상태:** Proposed

### 15. Trace와 Human Approval
